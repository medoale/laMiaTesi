# agentAnalysis

Runs three LLM agents (via [OpenRouter](https://openrouter.ai)) on the
vulnerability-fixing commits selected in
`../cveFixes/CVEfixes/Code/Data/repo_analysis_v2.csv`, under three different
information regimes, to measure **how much context an LLM needs to recognize
the vulnerability fixed by a commit**:

| agent | receives | repo access |
|---|---|---|
| `agent1` | `code_before` + `code_after` + `diff` of the commit | none |
| `agent2` | same code sections | full clone, navigable via tools |
| `agent3` | nothing | full clone, navigable via tools |

The delta between agent 1 and agent 2 measures the value of repository
context; agent 3 measures whether the vulnerability can be found with no
hint of where the fix is.

## Guarantees

- **Stateless by construction.** Every agent call builds a brand-new
  `messages` array containing only its own prompt: no agent can see another
  agent's conversation, or a previous commit. OpenRouter keeps no state
  between requests.
- **Blind prompts.** No repository names, file paths or CVE metadata in the
  prompt text — code sections are only numbered (`FILE 1`, `FILE 2`, …).
- **Sandboxed navigation.** Agents 2 and 3 explore the clone through two
  tools (`list_dir`, `read_file`) executed by our code, never by the model.
  Paths are resolved and jailed inside the repo; `.git` is invisible and
  unreadable — the history contains the fix commit itself and would leak
  the answer.
- **Pre-fix snapshot.** The clone is checked out at the **parent** of the
  fix commit: the vulnerability is still in the code. One clone per CSV row,
  shared by agents 2 and 3, always deleted afterwards (even on Ctrl-C).
- **Agent 1 cannot touch the repo.** Its API call advertises no tools at
  all, and its code path (`single_completion`) contains no tool execution.

## Files

```
agentAnalysis/
├── main.py      orchestrator: CSV + DB access, clone lifecycle, dispatch,
│                output files, resume. Contains filter_rows() (placeholder).
├── agent1.py    PROMPT_1 (placeholder) + run(api_key, changes)
├── agent2.py    PROMPT_2 (placeholder) + run(api_key, changes, repo_dir)
├── agent3.py    PROMPT_3 (placeholder) + run(api_key, repo_dir)
└── common.py    OpenRouter calls, tool schemas, sandboxed filesystem tools,
                 tool-calling loop, MODEL constant.
```

## Inputs

- `repo_analysis_v2.csv` — which (repo, commit, parent) rows to process
  (only `status == ok` rows; further selection goes in `filter_rows()`).
- `CVEfixes.db` — the `file_change` table provides before/after/diff per
  commit. **Complete only on the cluster** (the local copy is a simulation
  without `file_change`).

Both paths are resolved relative to the repository layout
(`../cveFixes/CVEfixes/Code/Data/`), so no path changes are needed on a
machine with the same structure.

## Outputs

- `agent_responses.jsonl` — raw log, one JSON line per agent call
  (`repo_url, commit, parent, agent, model, created_at, status, response`).
  Appended and flushed call by call.
- `results.jsonl` — consumable view, rebuilt from the raw log at the end of
  every run: one line per (repo, version), reference first, then the three
  results together:

  ```json
  {"repo_name": "owner/repo", "repo_url": "…",
   "commit": "<fix commit>", "parent": "<measured pre-fix version>",
   "agent1": {"status": "ok", "response": "…"},
   "agent2": {"status": "ok", "response": "…"},
   "agent3": null}
  ```

  Agents not yet run are `null`; when the log contains an error later
  retried successfully, the retry wins (last record per agent).

**Resume**: on restart, calls already answered with `status: "ok"` are
skipped; errors are retried. `results.jsonl` can be regenerated by hand with
`python3 -c "import main; main.build_grouped_results()"`.

## Before running

1. Write the three prompts (`PROMPT_1/2/3` in the agent files) and the row
   filter (`filter_rows()` in `main.py`).
2. Add the key to the ini file (path in the `CVEFIXES_INI` constant of
   `common.py` — adjust per machine):

   ```ini
   [OpenRouter]
   api_key = sk-or-v1-…
   ```

3. Pick a `MODEL` in `common.py` that **supports tool calling** (agents 2
   and 3 need it; most `:free` models do not) and has a context window large
   enough for whole before/after/diff sections — nothing is truncated.
4. Optional: a valid `[GitHub] token` in the same ini authenticates the
   clones; without it clones are anonymous.

Then, from this folder:

```bash
python3 main.py
```
